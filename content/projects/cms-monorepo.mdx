## 배경

백오피스 CMS가 도메인별로 분산된 멀티레포 구조였습니다. 공통 코드 중복과 개별 배포로 인한 운영 복잡도가 누적됐고, 도메인 간 UI 일관성도 무너지고 있었습니다.

## 기술적 배경

파트너센터에서 이미 Yarn Workspaces 기반 모노레포 환경을 사용해본 경험이 있었습니다. 빌드 캐시와 병렬 빌드 같은 빌드 성능 측면의 한계를 느꼈고, 신규 통합 CMS는 앱 수가 많아질 것을 예상해 Turborepo로 결정했습니다.

## 설계 원칙

가장 고민한 부분은 "무엇을 공통화하고 무엇을 독립으로 둘 것인가"였습니다. 기준선은 변경 빈도로 잡았습니다. 변경 빈도가 낮은 인프라성 코드(공통 UI, API 클라이언트, ESLint 설정)는 공통 패키지로, 도메인별로 자주 바뀌는 비즈니스 로직은 독립적으로 유지했습니다.

## 접근 방식

- 공통 UI, API 클라이언트, 타입, ESLint 설정을 내부 패키지로 분리
- 도메인별 독립성을 유지하면서 단일 저장소 기반 운영 환경 구성
- Spring Boot BFF의 반복 구현을 AI 코드 생성 규칙으로 표준화

## 인사이트

모노레포 설계의 핵심은 무엇을 공통화하고 무엇을 독립으로 둘지 결정하는 기준선이었습니다. 변경 빈도라는 기준이 의외로 강력한 가이드라인이 됐습니다.
