## 배경

기존 백오피스 CMS는 도메인별 멀티레포로 운영되고 있었습니다. 공통 UI와 API 클라이언트 패턴이 레포마다 반복됐고, 신규 CMS를 추가할 때마다 프로젝트 세팅과 배포 설정을 처음부터 다시 구성해야 했습니다.

기존 CMS를 한 번에 전면 이관하기보다는, 앞으로 추가될 신규 CMS를 한 저장소에서 수용할 수 있는 Turborepo 기반 모노레포를 설계했습니다.

## 기술적 배경

사내 CMS는 Spring Boot 서버가 인증, 메뉴, 권한, 정적 리소스 서빙을 담당하는 구조였습니다. 이미 정해진 운영 방식이었기 때문에, 프론트엔드 구조를 완전히 독립적으로 가져가기보다는 기존 CMS 서버 환경 안에서 안정적으로 추가될 수 있는 방식이 필요했습니다.

신규 통합 CMS는 여러 도메인 앱이 계속 추가될 구조였기 때문에, 앱별 독립성을 유지하면서도 공통 UI, API 클라이언트, 타입, ESLint 설정을 함께 관리할 수 있는 모노레포 구조를 선택했습니다.

## 설계 원칙

변경 빈도가 낮은 코드(공통 UI, API 클라이언트, 타입, ESLint 설정)는 공통 패키지로 분리하고, 도메인별로 자주 바뀌는 비즈니스 로직은 각 앱 내부에 유지했습니다.

또한 신규 Next.js 앱을 별도 서비스처럼 분리하기보다는, 기존 CMS의 URL 체계와 정적 파일 배치 방식에 맞춰 빌드 산출물이 자연스럽게 포함될 수 있도록 설계했습니다.

## 접근 방식

- Turborepo 기반으로 신규 CMS 앱을 수용하는 모노레포 구조 설계
- 공통 UI, API 클라이언트, 타입, ESLint 설정을 내부 패키지로 분리
- Next.js 앱별 독립성을 유지하면서 단일 저장소 기반 개발·빌드 흐름 구성
- 신규 CMS 추가 시 반복되는 프론트엔드 초기 세팅 패턴 정리

## AI 활용

신규 CMS를 추가할 때 반복되는 작업을 줄이기 위해 프로젝트 전용 AI 작업 규칙을 함께 정리했습니다.

하나는 앱이 추가될 때 필요한 Spring Boot 쪽 반복 구현(Controller, 라우팅 등)을 누락 없이 처리할 수 있도록 순서를 정리하였고,

추가로 UI 구현 시 앱의 의존성을 먼저 확인하고, 공통 UI 컴포넌트와 SCSS 토큰을 우선 사용하도록 만들었습니다. 버튼, 테이블 같은 공통 UI를 먼저 검토하도록 해서, 앱마다 다른 구현 방식이 늘어나는 것을 줄이는 것이 목적이었습니다.

마지막으로 티켓 단위 작업 흐름도 함께 정리했습니다. 작업 시작 전에는 요구사항, 디자인, API 스펙을 먼저 확인하도록 했고, 작업 완료 시에는 빌드·린트 통과 여부와 변경 범위가 의도한 범위 안에 있는지를 점검하도록 구성했습니다.

AI가 임의로 코드를 만들게 하기보다는, 기존 코드 구조와 사내 CMS 규칙을 먼저 읽고 그 안에서 반복 작업을 처리하도록 기준을 만든 것이 핵심이었습니다.

## 인사이트

모노레포 설계의 핵심은 모든 것을 한 번에 통합하는 것이 아니라, 정해진 사내 운영 환경 안에서 앞으로 반복될 신규 CMS 개발을 어떤 기준으로 흡수할지 정하는 것이었습니다. 변경 빈도가 낮은 인프라성 코드는 공통
화하고, 도메인별 비즈니스 로직은 앱 단위로 독립시키는 기준이 효과적이었습니다.
