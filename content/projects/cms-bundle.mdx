## 배경

통합 CMS는 여러 Next.js 앱을 빌드한 뒤 그 결과물을 Spring Boot 서버의 정적 자원으로 포함시켜 배포하는 구조였습니다. 신규 앱이 늘어나면서 프론트엔드 수정만으로도 여러 앱 빌드와 백엔드 JAR 빌드가 순차로 이어져 배포 시간이 크게 길어졌습니다.

## 분석

AI 코딩 도구를 활용해 빌드 파이프라인 전체를 분석했습니다. 10여 개 앱, 수천 개 파일 모노레포에서 병목을 직접 추적하기 어려웠기 때문입니다.

여러 Next.js 앱의 순차 빌드가 가장 큰 병목이었습니다. CI 환경에서는 너무 많은 앱을 한 번에 빌드하면 메모리 부족이나 일시적 실패가 발생할 수 있어, 단순 병렬화만으로는 부족하다는 점도 파악했습니다.

## 접근 방식

분석 결과를 바탕으로 단계별 개선을 진행했습니다.

- Turborepo 병렬 빌드를 도입하고, CI에서는 실패 시 더 낮은 병렬 수로 자동 재시도해 안정성 확보
- turbo와 Docker 레이어 캐싱으로 재빌드 범위 축소
- yarn PnP 로더 파일을 제거해 빌드 환경 단순화
- 빌드 전 디스크 여유가 부족하면 자동 정리하도록 구성해 CI 빌드 안정성 확보

## 결과

배포 시간은 14분에서 8분으로 줄었습니다. 빌드 속도뿐 아니라 실패 시 자동 재시도와 디스크 정리 흐름까지 정리해 배포 과정의 안정성도 함께 개선했습니다.

## 인사이트

수천 개 파일에 걸친 빌드 파이프라인 분석은 혼자 시작하기 어려운 작업이었지만, 문제를 먼저 정의하고 AI를 도구로 활용해 대규모 코드베이스에서도 단계적으로 실질적인 개선을 만들어낼 수 있었습니다.
